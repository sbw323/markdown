## Agent Framework Plan: LLM-Assisted Literature Review Pipeline Builder

### Architecture Overview

The agent follows a sprint-based architecture where Claude iterates through defined development sprints, using tools to write code, run tests, and validate outputs. Four core modules govern the agent's behavior:

```
lit_review_agent/
├── orchestrator.py      # Main loop: sprint sequencing, LLM calls, decision routing
├── sprints.py           # Sprint definitions: goals, acceptance criteria, file targets
├── prompts.py           # All system/user prompts for each agent action
├── tools.py             # Callable functions the agent can invoke (file I/O, shell, validation)
├── config.py            # API keys, paths, model selection, runtime flags
└── workspace/           # The agent builds the pipeline project here
    └── lit_review_pipeline/   # (generated by the agent)
```

### Module 2: `prompts.py`

This module holds every prompt template the agent uses. Prompts are keyed by purpose and use Python string formatting for injection of dynamic content.

**Prompt categories to define:**

1. **`SYSTEM_DEVELOPER`** — The persistent system prompt establishing the agent's role. Sets the persona as a senior Python developer building a scientific literature review pipeline. Specifies coding standards: type hints, docstrings, logging via `logging` module, no print statements, all file paths derived from `config.py`, error handling with informative messages.

2. **`SPRINT_INSTRUCTIONS`** — Template that receives the current sprint's `goal`, `acceptance_criteria`, and `files_to_produce`. Structures the request as: "You are executing sprint {id}: {name}. Your goal is: {goal}. You must produce the following files: {files}. The code must satisfy these acceptance criteria: {criteria}."

3. **`CONTEXT_INJECTION`** — Template wrapping existing file contents so the agent sees what it's building on. Format: "Here are the current project files relevant to this sprint:\n\n--- {filepath} ---\n{content}\n---". Populated from the sprint's `context_files` list plus any files from prior sprints.

4. **`CODE_GENERATION`** — The main generation prompt. Combines SYSTEM_DEVELOPER + CONTEXT_INJECTION + SPRINT_INSTRUCTIONS. Appends: "Return ONLY the file contents. Use the tag `<file path=\"{path}\">...contents...</file>` for each file you produce. Do not include any commentary outside the file tags."

5. **`VALIDATION_FIX`** — Used when a validation check fails. Template: "The following validation error occurred after sprint {id}:\n\n{error_output}\n\nHere is the code that caused it:\n\n{file_contents}\n\nFix the code to resolve this error. Return the complete corrected file using the same `<file>` tag format."

6. **`REVIEW_PROMPT`** — Optional self-review prompt. After code generation, the agent reviews its own output against acceptance criteria before writing files. "Review the following code against these acceptance criteria: {criteria}. List any criteria that are NOT met. If all are met, respond with 'ALL_PASS'."

Each prompt is a string constant with `{}` placeholders. No f-strings — the orchestrator calls `.format()` at runtime. This keeps prompts inspectable and editable without touching orchestrator logic.

### Module 3: `tools.py`

This module defines every concrete action the agent can take on the local filesystem and runtime environment. Each tool is a plain Python function that returns a standardized result dict: `{"success": bool, "output": str, "error": str | None}`.

**Tools to implement:**

1. **`write_file(path: str, content: str)`** — Writes content to the specified path, creating parent directories as needed. Returns success/failure.

2. **`read_file(path: str)`** — Reads and returns file contents. Used by the orchestrator to inject context files into prompts.

3. **`list_directory(path: str)`** — Returns the directory tree. Used for validation of scaffold sprint.

4. **`run_shell(command: str, timeout: int = 120)`** — Executes a shell command via `subprocess.run()`. Captures stdout and stderr. Used for `pip install`, running scripts, Docker commands. Timeout prevents hangs.

5. **`validate_python_syntax(path: str)`** — Runs `py_compile.compile(path, doraise=True)`. Returns success if no syntax errors.

6. **`validate_file_structure(expected_paths: list[str])`** — Checks that every path in the list exists on disk. Returns a list of missing paths if any.

7. **`run_python_script(path: str, timeout: int = 300)`** — Executes `python {path}` via `run_shell`. Returns stdout/stderr. Used for integration test sprint.

8. **`parse_file_tags(llm_response: str)`** — Parses the LLM's response to extract `<file path="...">...</file>` blocks. Returns a list of `(path, content)` tuples. This is a pure string operation, not an OS tool, but lives here because it's part of the tool interface the orchestrator calls.

9. **`install_dependencies(requirements_path: str)`** — Runs `pip install -r {requirements_path}`. Called after the scaffold sprint.

10. **`start_grobid()`** — Runs `docker run -d -p 8070:8070 lfoppiano/grobid:0.8.1` and waits for the health endpoint to respond. Returns success once GROBID is accepting requests.

### Module 4: `orchestrator.py`

This is the main loop. It is intentionally simple — a linear sprint executor with a retry mechanism.

**Logic flow:**

```
1. Load sprints from sprints.py
2. Initialize state: current_sprint_index = 0, completed = []
3. Start GROBID via tools.start_grobid()

4. FOR each sprint in sprints:
   a. Gather context:
      - Read all context_files listed in the sprint
      - Read all files produced by previously completed sprints
      - Deduplicate (context_files take precedence)
   
   b. Build prompt:
      - Assemble CODE_GENERATION prompt from prompts.py
      - Inject sprint goal, acceptance criteria, file targets, context
   
   c. Call Claude API:
      - Send assembled prompt to claude-sonnet-4-20250514
      - Temperature 0 for deterministic code generation
   
   d. Parse response:
      - Extract <file> tags via tools.parse_file_tags()
      - If no valid file tags found, retry from step (c) with a clarifying prompt (max 2 retries)
   
   e. Write files:
      - For each extracted file, call tools.write_file()
   
   f. Self-review (optional, configurable):
      - Send generated code + acceptance criteria to REVIEW_PROMPT
      - If response is not "ALL_PASS", feed review feedback back into VALIDATION_FIX and regenerate
   
   g. Validate:
      - Call the sprint's designated validation_tool
      - If validation passes: log success, append sprint to completed list, continue
      - If validation fails: feed error into VALIDATION_FIX prompt, regenerate, re-validate
      - Max 3 fix attempts per sprint before logging failure and halting
   
   h. Post-sprint hook:
      - After S0_scaffold: call tools.install_dependencies()
      - After S10_integration_test: log final report

5. Print summary: which sprints passed, which failed, total elapsed time
```

**Key design decisions for the orchestrator:**

- **No checkpointing** — as specified, the VM is stable. State is implicit in the filesystem: if `workspace/lit_review_pipeline/01_ingest.py` exists, S1 was completed. On re-run, the orchestrator can optionally skip sprints whose output files already exist (add a `--resume` flag).
- **Context window management** — later sprints accumulate more context files. If total context approaches the model's limit, the orchestrator truncates older sprint outputs to just their function signatures and docstrings (a "skeleton" mode). `tools.py` should include a `extract_skeleton(path: str)` helper that uses AST parsing to pull signatures and docstrings only.
- **Logging** — the orchestrator writes a structured log to `workspace/agent_log.jsonl`. Each entry records: sprint ID, timestamp, prompt token count, response token count, validation result, retry count. This lets you audit the agent's decisions after the fact.
- **Model selection** — `config.py` sets the model. Use `claude-sonnet-4-20250514` for all sprints during development. The agent builds the *pipeline's* config to use whichever model you choose for the literature review itself — these are separate concerns.

Run the agent with:

```bash
python orchestrator.py
```

It will print progress per sprint and halt on unrecoverable failures. The finished pipeline will be in `workspace/lit_review_pipeline/`, ready for you to drop PDFs into `data/pdfs/` and run the numbered scripts in order.
