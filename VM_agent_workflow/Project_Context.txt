## 1. Project Context

### 1.1 Objective

Design and implement a **Dynamic Step-Response Simulation** framework that systematically varies influent loading conditions and aeration curtailment levels across an ASM3 wastewater treatment benchmark model (Simulink steady-state solver) to produce a comprehensive training database for risk-energy optimization modeling.

### 1.2 What the Framework Produces

A structured dataset where each row represents one complete step-response cycle:

```
[Influent Conditions] × [Aeration Profile] → [Baseline] → [Curtailment Response] → [Recovery] → [Derived Metrics]
```

The database captures: how the plant responds to aeration curtailment under different loading conditions, how much energy is saved, how much effluent quality degrades, and whether the plant recovers to baseline after restoring nominal aeration.

### 1.3 Architecture — Responsibility Boundary

```
┌──────────────────────────────────────────────────────────────────┐
│  PYTHON LAYER — Orchestrator / CRUD / Data Analysis / ML         │
│                                                                  │
│  ✓ Influent percentile matrix generation (statistical sampling)  │
│  ✓ Aeration reduction profile grid generation (parameter sweep)  │
│  ✓ YAML config management                                        │
│  ✓ MATLAB engine invocation and sequencing                       │
│  ✓ Output data collection, assembly, and storage (CSV/Parquet)   │
│  ✓ Derived metric computation (damage indices, energy ratios)    │
│  ✓ Future: ML model training on assembled database               │
│                                                                  │
│  ✗ NEVER: ODE solving, biological kinetics, COD fractionation,   │
│           reactor mass balances, settling physics, or any        │
│           calculation that belongs to the ASM3 model             │
└──────────────────────────┬───────────────────────────────────────┘
                           │ file-based interface (.mat files in, .csv/.mat out)
┌──────────────────────────▼───────────────────────────────────────┐
│  MATLAB LAYER — Simulink Model Execution                         │
│                                                                  │
│  ✓ benchmarkinit / benchmarkss / stateset (model lifecycle)      │

│  ✓ Settler model physics                                         │
│  ✓ ASM3 biological process calculations                          │
│  ✓ COD fractionation and sCOD computation                        │
│  ✓ Aeration energy calculation                                   │
│  ✓ State variable capture and workspace management               │
│  ✓ Per-iteration data writing (settler output, energy, sCOD)     │
│                                                                  │
│  Pattern: clear all → reload from .mat → run → save → repeat     │
└──────────────────────────────────────────────────────────────────┘
```

**Critical boundary rule:** Every number that comes out of the Simulink model is produced by MATLAB. Python only performs arithmetic on MATLAB's outputs (ratios, differences, normalization) — never on raw state variables or biological parameters.

### 1.4 COD Fractionation — Boundary Clarification

The sCOD calculation (referenced in `perf_plant_LT_DR.m` line 129) uses ASM3 stoichiometric COD conversion factors applied to settler effluent state variables. This is a **model-coupled calculation** — the coefficients are ASM3 parameters, not generic arithmetic. Therefore:

- **sCOD computation stays in MATLAB.** The MATLAB data writer must output sCOD as a pre-computed value alongside settler concentrations.
- **Damage indices and energy ratios are computed in Python.** These are simple ratios of MATLAB outputs: `(experimental - nominal) / limit_or_nominal`. They require no model knowledge beyond the output values and regulatory limits.

## 2. The Step-Response Protocol

Each experimental cycle consists of three sequential steady-state simulations sharing the same influent condition. This is the atomic unit of the database.

```
│  STEP A:     │     │  STEP B:          │     │  STEP C:     │
│  BASELINE    │────▶│  CURTAILMENT      │────▶│  RECOVERY    │
│              │     │                    │     │              │
│  Nominal KLa │     │  Reduced KLa      │     │  Nominal KLa │
│  Influent Pₖ │     │  Same Influent Pₖ  │     │  Same Pₖ     │
│  → SS solve  │     │  → SS solve        │     │  → SS solve  │
│  → Baseline  │     │  → Experimental    │     │  → Recovery  │
│    outputs   │     │    outputs         │     │    outputs   │
```

**Step A — Baseline:** Initialize model with `benchmarkinit`, set nominal aeration (KLa1=KLa2=0, KLa3=KLa4=240, KLa5=110), load influent percentile vector into `exp_influentQ.mat`, run `benchmarkss` to convergence. Save state and record settler output as baseline.

**Step B — Curtailment:** From Step A's converged state, apply reduced KLa3/KLa4/KLa5 values. Run `benchmarkss` to convergence. Record settler output, energy, and sCOD as experimental values.

**Step C — Recovery:** From Step B's converged state, restore nominal KLa values. Same influent. Run `benchmarkss` to convergence. Record settler output as recovery values. Run `benchmarkinit` to clear workspace for next cycle.

### 2.1 Workspace Survival Pattern

The existing `ssASM3_DR_datagen.m` demonstrates the proven pattern for surviving `benchmarkinit`'s workspace destruction:

```matlab
while true
    clear all;                    % Nuclear option — wipe everything
    load('sim_config.mat');       % Restore immutable experiment config
    load('sim_state.mat');        % Restore mutable iteration state
    if iter > num_iterations      % Check completion
        break;
    end
    benchmarkinit;                % Initialize model (clears workspace again)
    load('sim_config.mat');       % Restore AGAIN after benchmarkinit
    load('sim_state.mat');        % Restore AGAIN
    % ... extract parameters, configure model, simulate, save data ...
    iter = iter + 1;
    save('sim_state.mat', 'iter');
end
```

This pattern MUST be preserved in the refactored script. The two-phase reload (before and after `benchmarkinit`) is not redundant — it is the core robustness mechanism.

### 2.2 Convergence & StopTime Strategy

The existing code scales StopTime with aeration reduction severity (200 iterations at nominal → 2000 iterations at heavy reduction). This reflects the physical reality that more reduced aeration requires longer to reach a new steady state.

**Proposed approach:** Define StopTime as a function of the maximum aeration reduction factor:

```
reduction_factor = min(KLa3/KLa3_nom, KLa4/KLa4_nom, KLa5/KLa5_nom)
StopTime = base_time + scale_factor × (1 - reduction_factor)
```

Where `base_time` (for nominal) and `scale_factor` (for maximum reduction) are config parameters. The existing code suggests `base_time ≈ 200` and the maximum around `2000` at ~57% reduction of the most-reduced parameter.

## 3. Experiment Parameter Space

### 3.1 Influent Variability — Percentile Matrix

**Source data:** `ASM3_Input` matrix — 21 columns (col 1 = time, cols 2–14 = water quality, col 15 = Q, col 16 = temperature, cols 17–21 = dummy variables).

**Sampling strategy:**

1. Compute percentiles (10th, 25th, 50th, 75th, 90th) of the Q vector (column 15).
2. For each Q percentile, identify the timesteps where Q falls nearest to that percentile value.
3. At those timesteps, extract the corresponding water quality parameters (cols 2–14) and temperature (col 16).
4. Average across the identified timesteps to produce one representative influent vector per Q percentile.
5. Format each vector to match `CONSTINFLUENT.mat` structure (2-row matrix with identical rows, 21 columns).

**Output:** 5 influent condition files, one per percentile, each in `CONSTINFLUENT.mat` format.

**Responsibility:** Python generates these files. No biological interpretation — purely statistical sampling and matrix formatting.

### 3.2 Aeration Reduction Profiles

**Nominal values** (from experiment plan): KLa3 = 240, KLa4 = 240, KLa5 = 110.

**Reduction constraint:** Minimum 50% of nominal for each parameter.

**Reduction structure:** The existing code applies a proportional reduction across all three tanks (KLa3 and KLa4 share a factor; KLa5 tracks proportionally at its own nominal). The experiment plan says "Numerous different reduced aeration profiles" which suggests systematic coverage of the reduction space.

**Proposed grid — Proportional reduction with a single factor:**

```
reduction_factors = [1.0, 0.95, 0.90, 0.85, 0.80, 0.75, 0.70, 0.65, 0.60, 0.55, 0.50]

For each factor f:
    KLa3 = 240 × f
    KLa4 = 240 × f
    KLa5 = 110 × f
```

This yields 11 aeration profiles (including the nominal baseline at f=1.0).

**Open question for review:** Should KLa3/KLa4/KLa5 vary independently? Independent variation of 3 parameters at 11 levels would produce 1,331 profiles — likely excessive for steady-state simulation. The proportional approach covers the reduction axis with 11 profiles. A compromise (e.g., vary KLa5 independently while keeping KLa3=KLa4 tied) could be a Phase 2 extension.

### 3.3 Total Experiment Size

```
5 influent percentiles × 11 aeration profiles × 3 steps per cycle = 165 steady-state simulations
```

Each simulation produces one row of settler output + energy + sCOD. The assembled database has 55 complete cycles (5 × 11), each with baseline/curtailment/recovery triplets.

**Responsibilities:**

```python
def compute_damage_indices(
    baseline: dict,
    experimental: dict,
    limits: dict,
) -> dict:
    """Compute effluent damage indices.
    
    COD_dmg = (COD_exp - COD_nom) / COD_limit
    NH4_dmg = (NH4_exp - NH4_nom) / NH4_limit
    
    Inputs are settler effluent values from MATLAB.
    Limits come from config (regulatory_limits).
    """

def compute_energy_reduction(
    baseline_energy: float,
    experimental_energy: float,
) -> float:
    """Compute fractional energy reduction.
    
    E_red = (E_exp - E_nom) / E_nom
    
    Negative value = energy saved during curtailment.
    """
```

### 5.6 MATLAB — Step-Response Engine (`ssASM3_DR_datagen.m`)

**Responsibilities:**

- Read experiment parameters from `sim_config.mat`
- Execute the 3-step protocol (baseline → curtailment → recovery) per cycle
- Survive `benchmarkinit` workspace destruction using the `clear all` + file-reload pattern
- Write per-step outputs via `ssData_writer_reliability`

**Refactoring from existing code:**

The existing script runs a flat loop of independent steady-state simulations. The refactored version must implement the **3-step protocol** where Steps B and C inherit state from prior steps within a cycle.

**Structural design:**

```matlab
%% OUTER LOOP: One iteration per experiment cycle
while true
    clear all;
    
    % Reload state
    load('sim_config.mat');   % Contains: test_case, influent_vector, nominal_kla,
                              %           output_file, ss_model, cycle_id, ...
    load('sim_state.mat');    % Contains: step (1=baseline, 2=curtailment, 3=recovery),
                              %           cycle_complete flag
    
    if cycle_complete
        break;
    end
    
    %% --- STEP A: BASELINE (step == 1) ---
    if step == 1
        benchmarkinit;
        load('sim_config.mat');
        load('sim_state.mat');
        
        % Set nominal KLa values
        % Load influent percentile into CONSTINFLUENT
        % Configure and run benchmarkss
        % stateset → capture baseline settler output
        % ssData_writer_reliability(..., 'baseline')
        % Save workspace state for Step B inheritance
        
        step = 2;
        save('sim_state.mat', 'step', 'cycle_complete');
    end
    
    %% --- STEP B: CURTAILMENT (step == 2) ---
    if step == 2
        % Load Step A's converged state (NO benchmarkinit — preserve biology)
        load('workspace_baseline.mat');
        load('sim_config.mat');
        load('sim_state.mat');
        
        % Override KLa to reduced values
        % Configure StopTime for convergence at reduced aeration
        % Run benchmarkss
        % stateset → capture curtailment settler output
        % ssData_writer_reliability(..., 'curtailment')
        % Save workspace state for Step C inheritance
        
        step = 3;
        save('sim_state.mat', 'step', 'cycle_complete');
    end
    
    %% --- STEP C: RECOVERY (step == 3) ---
    if step == 3
        % Load Step B's converged state (NO benchmarkinit — preserve biology)
        load('workspace_curtailment.mat');
        load('sim_config.mat');
        load('sim_state.mat');
        
        % Restore nominal KLa values
        % Same influent, same state
        % Run benchmarkss
        % stateset → capture recovery settler output
        % ssData_writer_reliability(..., 'recovery')
        
        cycle_complete = true;
        save('sim_state.mat', 'step', 'cycle_complete');
    end
end

% Cleanup temp .mat files
```

**Critical design decisions in the MATLAB script:**

1. **`benchmarkinit` is called ONLY in Step A.** Steps B and C load the prior step's converged workspace directly. This is essential — calling `benchmarkinit` between steps would destroy the accumulated biological state and defeat the purpose of the step-response protocol.
    
2. **The `clear all` + reload pattern is preserved** at the top of the while loop for robustness, but `benchmarkinit` is gated to Step A only.
    
3. **State inheritance:** Step A saves `workspace_baseline.mat` after convergence. Step B loads it, applies reduced KLa, re-runs, then saves `workspace_curtailment.mat`. Step C loads that, restores nominal KLa, re-runs.
    
4. **StopTime varies by step:** Baseline uses `base_stoptime`. Curtailment uses the scaled StopTime. Recovery uses `base_stoptime` (returning to nominal should converge quickly from a nearby state).
    
5. **Per-step data writing:** `ssData_writer_reliability` receives a step label ('baseline', 'curtailment', 'recovery') and includes it in the output row.

## 8. Decision Log

|Decision|Rationale|
|---|---|
|sCOD and energy calculated in MATLAB|These require ASM3 stoichiometric coefficients — model-coupled calculations|
|Damage indices and energy ratios in Python|Simple arithmetic on MATLAB outputs; no model knowledge needed|
|`benchmarkinit` gated to Step A only|Steps B/C must inherit biological state from prior steps|
|Proportional aeration reduction (single factor)|Matches existing code pattern; independent variation is combinatorially expensive|
|StopTime scaled linearly with reduction|Physical: reduced aeration = slower convergence. Config-driven.|
|`sim_config.mat` as Python→MATLAB interface|Avoids YAML parsing in MATLAB; Python writes, MATLAB reads|
|`clear all` + file-reload pattern preserved|Battle-tested workspace survival; the existing code proves it works|
|5 percentiles × 11 reduction levels = 55 cycles|Defensible coverage without excessive computation|
|Per-step data writing in MATLAB|Each step appends to CSV immediately; crash recovery loses only current step|
